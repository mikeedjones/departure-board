Lesson 4 · HTTP & urequests

Your first HTTP request

Lesson 3 got the board onto the network — associated, DHCP'd, holding a real IP. That connection is just a wire now; nothing has flowed over it yet. This lesson makes the first thing flow: a plain HTTP GET, out and back, with a JSON body parsed on the other end. That's the exact mechanism the departure board will eventually poll on a timer — this lesson proves it works before wiring it to a real transit API.

Two ways to ask for a page

"You can take a low-level approach to HTTP requests using raw sockets, or a high-level approach using the urequests library." Raspberry Pi Ltd, Connecting to the Internet with Raspberry Pi Pico W-series, §3.8

Sockets show you what HTTP actually is. urequests is what you'll actually use. Worth seeing both, in that order.

Raw sockets — HTTP with nothing hidden

# Get IP address for google.com
import socket
ai = socket.getaddrinfo("google.com", 80)
addr = ai[0][-1]

# Create a socket and make a HTTP request
s = socket.socket()
s.connect(addr)
s.send(b"GET / HTTP/1.0\r\n\r\n")

# Print the response
print(s.recv(512))
§3.8.1 "HTTP with sockets" Raspberry Pi Ltd, Connecting to the Internet with Raspberry Pi Pico W-series

getaddrinfo is DNS — turning "google.com" into an IP address, the same lookup your laptop does. socket.socket() + connect(addr) is the TCP handshake, same as SPI's chip-select-then-clock but over a network instead of a bus. Then send() writes literal bytes — b"GET / HTTP/1.0\r\n\r\n" is the entire HTTP request. No object, no framework: a method, a path, a version, and a blank line.

Why the double \r\n: HTTP headers are one per line, each ending \r\n. A request can have zero headers, so the protocol needs an unambiguous "that's all of them" marker — a line that's empty. Second \r\n with nothing before it is exactly that. recv(512) then hands back whatever the server sends: status line, headers, and body, all as one undifferentiated blob of bytes. Nothing parses it for you at this layer.
In b"GET / HTTP/1.0\r\n\r\n", what does that second, empty \r\n line signal to the server?

urequests — the same thing, wrapped

"It is much simpler to use the urequests library to make an HTTP connection." Raspberry Pi Ltd, Connecting to the Internet with Raspberry Pi Pico W-series, §3.8.2 
import urequests
r = urequests.get("http://www.google.com")
print(r.content)
r.close()

Same DNS lookup, same socket, same raw bytes underneath — urequests just does all of it and hands you back a response object instead of a byte soup. It follows redirects automatically, and it parses JSON for you if you ask:

>>> r = urequests.get("http://date.jsontest.com")
>>> r.json()
{'milliseconds_since_epoch': 1652188199441, 'date': '05-10-2022', 'time': '01:09:59 PM'}

That exact endpoint from the guide is dead now — date.jsontest.com doesn't resolve any more. The workbook below uses httpbin.org/json instead, which is still alive. Public test APIs rot; worth knowing before assuming a tutorial's URL still works.

Must call .close(): straight from the guide — "You must close the returned response object after making a request using the urequests library using response.close(). If you do not, the object will not be garbage-collected, and if the request is being made inside a loop this will quickly lead to a crash." A departure board polls forever. Skipping .close() is the difference between running for a day and running for five minutes before it falls over.

Looking ahead: reconnecting after a failed request

§3.8.3 extends lesson 3's connect-and-poll habit past startup: wrap the request in try/except, and on failure, re-check wlan.status() before deciding whether to reconnect. Not built into this lesson's workbook — one request, once, is enough to prove the mechanism — but it's captured in the reference doc for when the polling loop gets built for real.

Step — install urequests, make the request

urequests isn't guaranteed to already be on the device's filesystem, so install it first — this runs on your computer, fetches over your machine's network, then copies the result to the Pico over USB (the device doesn't need Wi-Fi for this step):

uv run mpremote mip install urequests

The workbook's main.py is lesson 3's connect sequence, unchanged, plus one HTTP GET against http://httpbin.org/json — a small, stable, no-auth JSON test endpoint — with the response parsed and closed:

url = "http://httpbin.org/json"
response = urequests.get(url)
try:
    print(f"status = {response.status_code}")
    body = response.json()
    print(f"slideshow title = {body['slideshow']['title']}")
finally:
    response.close()

The try/finally is deliberate: it guarantees .close() runs even if .json() or the dict lookup blows up, not just on the happy path — the same "don't just trust the happy path" instinct from lesson 3's status poll, applied here.

Deploy and reset:

uv run mpremote cp workbooks/lesson-0004-first-http-request/firmware/config.py :config.py
uv run mpremote cp workbooks/lesson-0004-first-http-request/firmware/main.py :main.py
uv run mpremote reset

vs. Arduino/C: this exact guide has no C-side equivalent of §3.8 — the C SDK chapter stops at "getting online," no HTTP client section. There's no urequests.get() analogue in the Pico SDK at all; you'd reach for lwIP's own HTTP client code directly, callback-driven rather than a single blocking-looking call. urequests buying you a one-line GET is a real, not cosmetic, MicroPython advantage here.
