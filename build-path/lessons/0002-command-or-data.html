Lesson 2 · SPI & the ILI9341

Command or data? How the display tells them apart

The LED lesson proved Python is really running on the chip. This one wires up the actual screen — a 240×320 ILI9341 TFT, 8-pin SPI breakout (no MISO broken out). Five wires, one vendored driver, and one mechanism worth understanding on purpose rather than absorbing by osmosis: how a single data line carries both display commands and display pixel data without the two getting confused.

"Breakout"? The ILI9341 is a bare driver-chip die, with pins far too fine and densely packed to hand-solder or breadboard. A breakout board is a small PCB that chip is pre-soldered onto, which "breaks out" those pins to a row of ordinary 0.1" header pins you can jump wires to. The screen itself is glued to the same board. Nothing about SPI requires a breakout — it's purely a packaging convenience so hobbyists don't need reflow-soldering equipment.

Step 1 — wire it up

No wiring mechanics here — you know how to seat a breadboard. Just the pin table, since which GPIO does what is a protocol decision, not a generic electronics one. Your board's silkscreen uses its own short names for standard SPI signals — the table below maps them:

GP17–19 are the RP2350's default hardware SPI0 pins. GP20–22 are just free GPIOs — DC, RESET, and BL aren't SPI signals at all. BL is a plain digital pin here, not tied to 3V3, so the backlight is software-controlled: on for good in this lesson, but nothing stops a later lesson from adding PWM dimming on the same wire. "DIN"/"CLK"/"BL" are just this breakout's naming for MOSI/SCK/backlight — same signals as the SPI glossary uses.
The win, in one sentence
You'll be able to point at every wire above and say exactly what electrical event it produces, and trace that straight through to the two driver methods that do all the work.

SPI, compressed

SPI is a synchronous serial bus with one host (here, the Pico — SPI calls it "master") and one or more peripherals ("slaves"). "Synchronous" means there's a shared clock line, SCK, generated by the host — every bit moves on a clock edge, so there's no baud-rate guessing the way there is on UART. Everything else about the bus is built from that one idea.

The confusing part is almost always the two data wires, because SPI is full duplex: it can shift a bit host→peripheral and a bit peripheral→host on the same clock edge, over two separate, one-directional wires. Neither wire is ever bidirectional — that's the piece easy to miss coming from something like I²C, where one wire (SDA) carries traffic both ways. Both SPI wire names are written from the host's point of view, which is the whole trick to keeping them straight:

WireName expands toDirection
MOSIMaster Out, Slave Inhost → peripheral
MISOMaster In, Slave Outperipheral → host

Read each name as "what does the master do on this wire": MOSI, the master sends out; MISO, the master takes in. CS (chip select) is a third, separate wire the host drives low to say "the next clock pulses are for you specifically" — needed because SCK/MOSI/MISO can be shared across several peripherals on one bus, and without CS every device would try to answer every clock pulse at once.

"[SPI] uses four logic signals... to support full duplex communication." Wikipedia — Serial Peripheral Interface

Those four are SCK, MOSI, MISO, CS — the "four-wire" bus. This display only ever receives (there's no touch controller, no status byte you read back, nothing to report), so the MISO half of full duplex has no job here. That's not a limitation of SPI, it's this specific device declining to use half of what the bus offers — which is also exactly why the breakout doesn't bother bringing an SDO/MISO pin out to a header at all.

Hypothetically, if you wanted the ILI9341 to send its chip-ID byte back to the Pico over SPI, which of the four SPI wires would carry that byte?

The problem: one wire, two meanings

Every byte sent to the ILI9341 is either a command (e.g. "set the active drawing window") or data (the parameters for that command, or raw pixel colors). MOSI is just a wire — it can't say which kind just arrived. The datasheet gives two answers to this:

ModeCommand/data signalingWires
3-lineA 9th bit before every byte, same data lineCS, SCK, SDA
4-line (what's wired above)A dedicated pin held high/low for the whole byteCS, SCK, SDA, D/CX
"The 3-line serial mode consists of the chip enable input (CSX), the serial clock input (SCL) and serial data Input/Output (SDA)... The 4-line serial mode consists of the Data/Command selection input (D/CX), chip enable input (CSX), the serial clock input (SCL) and serial data Input/Output (SDA)." ILI9341 Datasheet, ILI Technology Corp, v1.11, §7.1.8 "Serial Interface", p.33

D/CX is the DC pin on GP20 above. It isn't a generic SPI signal — it's an ILI9341-specific escape hatch bolted onto the bus, purely to answer "command or data?" without paying the 9-bits-per-byte tax of 3-line mode.

Step 2 — vendor the driver, deploy, run

The workbook has firmware/lib/ili9341.py, vendored unmodified from rdagger/micropython-ili9341 (MIT), plus config.py (the pin table above, as constants) and a short main.py. Note that main.py drives BL itself, as a plain Pin.OUT — the backlight has nothing to do with the ILI9341's SPI interface or its driver, it's just power to an LED that happens to sit behind the glass. Deploy the whole tree:

uv run mpremote mkdir :lib
uv run mpremote cp workbooks/lesson-0002-command-or-data/firmware/lib/ili9341.py :lib/ili9341.py
uv run mpremote cp workbooks/lesson-0002-command-or-data/firmware/config.py :config.py
uv run mpremote cp workbooks/lesson-0002-command-or-data/firmware/main.py :main.py
uv run mpremote reset
Why not cp -r: mpremote 1.28.0 has a bug where cp -r on a flat directory (no subdirectories) copying to a destination that doesn't exist yet on the device silently misfires — it parses -r correctly but its internal file-listing logic still concludes "no directories found" and falls back to a single-file copy, which then fails with a misleading "-r not specified" error. Individual cp calls sidestep it entirely, and lib/ is flat anyway. LICENSE-ili9341.txt stays in the repo only — nothing on the device imports it.
Win condition
Screen goes navy, "Departure Board" appears in white. That's real pixels, driven by SPI you can now trace wire-by-wire.

Watch the mechanism in the code you just ran

Every call in that driver — clear(), draw_text8x8(), all of it — bottoms out in two methods that do nothing but flip the DC pin:

def write_cmd_mpy(self, command, *args):
    self.dc(0)
    self.cs(0)
    self.spi.write(bytearray([command]))
    self.cs(1)
    if len(args) > 0:
        self.write_data(bytearray(args))

def write_data_mpy(self, data):
    self.dc(1)
    self.cs(0)
    self.spi.write(data)
    self.cs(1)

Flip DC, toggle CS around a raw spi.write(). Nothing more.

vs. Arduino: the DC-toggle trick is identical if you've used an ILI9341 library there before. What's different: CS is toggled by hand around every single write, in plain Python, on every command and every data burst — there's no SPI.beginTransaction()/endTransaction()-style wrapper managing it. MicroPython's machine.SPI is a thinner abstraction: it shifts bytes and nothing else.
The breakout has no MISO/SDO pin at all. Based on write_cmd_mpy and write_data_mpy, why is that safe to skip?
