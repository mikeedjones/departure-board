Lesson 2 · SPI & the ILI9341

Command or data? How the display tells them apart

You just wired a Pico 2 W to a 240×320 ILI9341 TFT over SPI, deployed firmware/main.py, and (hopefully) watched "Departure Board" render on the screen. That worked because of one specific mechanism worth understanding on purpose, rather than absorbing by osmosis: how a single serial data line carries both display commands and display pixel data, without them getting confused.

The win, in one sentence
You'll be able to point at every wire in your wiring table and say exactly what electrical event it produces, and trace that straight through to the two methods in the vendored driver that do all the work.

SPI, compressed

SPI (Serial Peripheral Interface) is a synchronous serial bus: one device (here, the Pico) generates a clock signal (SCK), and shifts bits out one at a time on a data line (MOSI, "master out slave in"), one bit per clock edge, MSB first. A chip-select line (CS) tells the peripheral when a stream of bits is actually meant for it, since multiple SPI devices can share the same clock and data lines.

"[SPI] uses four logic signals... to support full duplex communication. It is sometimes called a four-wire serial bus." Full duplex needs a second data line back to the host (MISO) — our display never talks back, so that wire simply isn't wired up. Wikipedia — Serial Peripheral Interface

That's the generic bus. It says nothing about what the bytes flowing over it mean — that's entirely up to the chip on the other end. Which is where it gets specific to the ILI9341.

The problem: one wire, two meanings

Every byte you send the ILI9341 is either a command (e.g. "set the active drawing window") or data (e.g. the parameters for that command, or raw pixel colors). MOSI is just a wire — it can't tell the chip which kind of byte just arrived. The ILI9341 datasheet gives two different answers to this problem, and your wiring table picked one of them for you.

ModeHow command vs. data is signaledWires needed
3-line serialA 9th bit (D/CX) is squeezed in before every 8-bit byte, on the same data lineCS, SCK, SDA
4-line serial (what we're using)A dedicated hardware pin (D/CX) is held high or low for the whole byteCS, SCK, SDA, D/CX
"The 3-line serial mode consists of the chip enable input (CSX), the serial clock input (SCL) and serial data Input/Output (SDA)... The 4-line serial mode consists of the Data/Command selection input (D/CX), chip enable input (CSX), the serial clock input (SCL) and serial data Input/Output (SDA)." ILI9341 Datasheet, ILI Technology Corp, v1.11, §7.1.8 "Serial Interface", p.33

That D/CX signal is exactly the DC pin in your wiring table, on GP20. It's not a generic SPI signal at all — it's an ILI9341-specific escape hatch, bolted onto the bus, purely to answer "is this byte a command or data?" without paying the 9-bits-per-byte tax of the 3-line mode.

Watch it happen in the code you deployed

firmware/lib/ili9341.py (vendored from rdagger/micropython-ili9341) has two methods that do nothing but flip that DC pin: 

def write_cmd_mpy(self, command, *args):
    self.dc(0)          # DC low  -> "the next byte is a command"
    self.cs(0)          # select the chip for this transaction
    self.spi.write(bytearray([command]))
    self.cs(1)           # deselect -> transaction over
    if len(args) > 0:
        self.write_data(bytearray(args))

def write_data_mpy(self, data):
    self.dc(1)          # DC high -> "these bytes are data"
    self.cs(0)
    self.spi.write(data)
    self.cs(1)

That's the whole mechanism. display.clear(), draw_text(), everything the driver does, bottoms out in these two calls: flip DC, toggle CS around a raw spi.write().

Note for the Arduino move: if you've used an Arduino ILI9341 library before, this will look familiar — the DC-pin-toggle trick is identical there. What's different here is that CS is toggled by hand around every single write (self.cs(0) / self.cs(1)), in plain Python, on every command and every data burst — there's no SPI.beginTransaction()/endTransaction()-style wrapper managing it for you. MicroPython's machine.SPI is a thinner abstraction: it does the byte shifting, and nothing else.
