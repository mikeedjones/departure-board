Lesson 3 · Wi-Fi & network.WLAN

Turning on Wi-Fi, and joining your network

Screen's proven. Now the second half of the mission starts: getting this board on the network so it can eventually pull real departure data. This lesson is deliberately narrow — just network.WLAN, powering up the radio, and joining an access point as a client. No HTTP yet, no JSON, just the handshake that everything else sits on top of.

Same chip as the LED: Lesson 1's LED quirk — Pin("LED") routing through the CYW43439 instead of the RP2350 — was foreshadowing this. The CYW43439 is the wireless chip; the RP2350 talks to it over its own internal SPI link. Wi-Fi and the on-board LED are the same piece of silicon, which is why both needed the same underlying firmware brought up before they'd respond.

Station mode vs. access point mode

network.WLAN can put the CYW43439 into either of two roles. This project only ever needs one of them:

InterfaceRoleUsed here?
network.STA_IFStation — join an existing access point as a client, same as a laptop joining Wi-FiYes
network.AP_IFAccess point — the board becomes the Wi-Fi network other devices joinNo — out of scope

The connect sequence

Straight from the Pico W-series networking guide:

import network

wlan = network.WLAN(network.STA_IF)
wlan.active(True)
wlan.connect('Wireless Network', 'The Password')

while not wlan.isconnected() and wlan.status() >= 0:
    print("Waiting to connect:")
    time.sleep(1)

print(wlan.ifconfig())
"We're using the network library to talk to the wireless hardware." Raspberry Pi Ltd, Connecting to the Internet with Raspberry Pi Pico W-series, §3.6

Three calls do all the work. WLAN(network.STA_IF) just constructs a handle — nothing happens on the chip yet. active(True) is the one that actually loads the CYW43439's Wi-Fi firmware and powers the radio; the guide notes elsewhere that you have to call it before even reading the chip's MAC address, because nothing on the chip responds until firmware is loaded. connect(ssid, password) kicks off association and returns immediately — it doesn't block until you're online, it just tells the chip to start trying.

Non-blocking is the whole reason for the poll loop: because connect() returns instantly, your code has to ask the chip "are we there yet?" itself, in a loop, checking a status value each time. There's no callback, no async/await here — it's a plain while loop with a manual counter.

Reading the status codes

The workbook's version (below) polls wlan.status() instead of isconnected(), because the numeric status distinguishes why it isn't connected yet. The values are the CYW43 driver's own link-status enum, passed straight through to MicroPython:

ValueConstantMeaning
0CYW43_LINK_DOWNNo connection, no activity
1CYW43_LINK_JOINAssociated with the AP, no IP yet
2CYW43_LINK_NOIPJoined, waiting on DHCP for an IP
3CYW43_LINK_UPConnected — has an IP
-1CYW43_LINK_FAILFailed, other reason
-2CYW43_LINK_NONETNo matching access point found
-3CYW43_LINK_BADAUTHWrong password, auth rejected
"The values returned by the wlan.status() call are defined in the CYW43 wireless driver, and are passed directly through to user-code." Raspberry Pi Ltd, Connecting to the Internet with Raspberry Pi Pico W-series, §3.6.1

That's why the more robust poll loop (also in the guide, and what the workbook uses) checks status < 0 or status >= 3 to decide when to stop: negative means it's dead and won't recover on its own, 3 means success, and 0/1/2 are all "still in progress, keep waiting."

Your board has successfully associated with the router and negotiated encryption, but hasn't received an IP address from DHCP yet. What status code is that?

Step — fill in credentials, deploy

The workbook has firmware/config.py (gitignored — same pattern as lesson 2's pin config, just now holding a secret instead of pin numbers) and firmware/main.py. Open config.py and replace the two placeholder values with your actual network name and password:

Which single method call is the one that actually powers up the CYW43439's Wi-Fi firmware — distinct from constructing the WLAN handle or calling connect()?
If it won't associate on a network you know is good: the wireless country defaults to an unset, conservative worldwide-safe channel set, which can exclude channels your specific router uses. import rp2; rp2.country('GB') (using your own country's code) is the guide's fix if this ever bites — not needed here since it connected, but worth knowing before it's the reason something silently fails to join later.
vs. Arduino/C: the equivalent C SDK sequence folds country, station-mode, and connect-with-timeout into three explicit calls, with the timeout as a parameter rather than a loop you write: 
cyw43_arch_init_with_country(CYW43_COUNTRY_UK);
cyw43_arch_enable_sta_mode();
cyw43_arch_wifi_connect_timeout_ms(ssid, pass, CYW43_AUTH_WPA2_AES_PSK, 10000);
MicroPython gives you the same three ideas — country, mode, connect — but as separate, inspectable steps: active(True) does what cyw43_arch_init_with_country() + enable_sta_mode() do together, and the poll loop replaces the built-in timeout with one you control yourself.
