Lesson 1 · Toolchain & firmware layout

A tour of firmware/, before you've flashed anything

Right now the Pico is plugged into USB and nothing else has happened — no MicroPython on it, no screen wired up. Good moment to stop and understand what all the code already sitting in firmware/ is actually going to do to that board, before you do it.

The win, in one sentence
You'll know the difference between the two completely different things people call "uploading code to a microcontroller" — and be able to look at any file in firmware/ and say which of the two it belongs to.

What's on the board right now

A brand-new Pico 2 W's flash memory holds whatever the factory (or a previous owner) put there — typically nothing runnable, or a small demo. Plugging it into USB just powers it on and lets your computer talk to whatever bootloader or program is currently in that flash. It does not put any of your code on it. That's a separate, deliberate step, and — this is the part worth slowing down for — it's actually two different steps that are easy to conflate if you're coming from Arduino.

Two different kinds of "upload"

On Arduino, there's one mental model: you write a sketch, hit Upload, and the whole thing — your code plus the C runtime it needs — gets compiled and flashed onto the chip as one unit. Every change means a full reflash.

MicroPython splits that into two layers that don't have an Arduino equivalent:

What it isHow oftenHow
1. Flash the interpreter The MicroPython runtime itself — a full firmware image (.uf2) that replaces everything on the chip's flash Rarely — once, or when upgrading MicroPython versions Hold BOOTSEL while plugging in USB, the Pico mounts as a USB drive, drop the .uf2 file onto it
2. Deploy your code Plain .py files, copied onto a small filesystem that the already-running interpreter reads and executes Every time you change a line of code scripts/deploy.sh, via mpremote — no BOOTSEL involved

This is the whole reason MicroPython iterates faster than Arduino once it's set up: step 1 happens once, step 2 is just a file copy over USB serial, no compilation. You haven't done step 1 yet — that's the very next thing, per README.md.

What happens when the interpreter takes over: boot.py, then main.py

Once MicroPython is flashed and running, it looks for two specific filenames on its filesystem, in a fixed order, every time it starts:

"_boot.py ... boot.py ... If boot.py is found then it is executed. You can add code in boot.py to perform custom one-off initialisation ... a file named main.py ... is executed next ... main.py is for any Python code that you want to run each time your device starts." MicroPython docs — Reset and boot

That's the direct equivalent of Arduino's setup() then loop() — except here they're two separate files rather than two functions in one file, and there's no automatic re-looping: main.py runs top to bottom once (our main.py doesn't loop at all yet — it draws one screen and stops).

Right now firmware/boot.py is deliberately just a docstring — there's no one-off init this project needs yet. It exists as the right place for that, later (things like configuring the filesystem or handling a "hold this button to skip main.py" recovery pattern typically live there).

The rest of firmware/, file by file

firmware/
  boot.py             <- runs first, currently empty on purpose
  main.py             <- runs second; the actual hello-world program
  config.example.py   <- template for pin numbers (see below)
  lib/
    ili9341.py          <- vendored driver, third-party (MIT-licensed)
    xglcd_font.py        <- font rendering the driver depends on
    fonts/Unispace12x24.c <- one bitmap font, also vendored

Why lib/ is vendored code, not a package install: MicroPython has a real package manager — mip, conceptually pip's counterpart, installing from micropython-lib or straight from a GitHub URL. For this project the driver was copied directly into firmware/lib/ instead, so the exact vendored version is committed and reviewable in this repo rather than fetched fresh (and potentially changed) on every deploy. mip is worth knowing about for other libraries later.

Why config.example.py → config.py: this is the same template/local-override split you already know from .env.example → .env in other ecosystems. config.py holds your actual pin numbers, is gitignored, and is created automatically (copied from the example) the first time you run scripts/deploy.sh — see the script itself for that.

What main.py does is the subject of the next lesson — it talks to the ILI9341 over SPI, and that deserves its own lesson rather than a rushed paragraph here. This one stops at "it exists and gets deployed"; lesson 2 opens it up.

What scripts/deploy.sh is actually doing

It's a thin wrapper around mpremote commands you could type by hand — nothing in it is magic:

  1. Checks mpremote is installed (that's the tool talking to the board over USB serial)
  2. Copies config.example.py to config.py if you haven't already customized it
  3. Creates lib/ and lib/fonts/ on the device's filesystem — this is a separate, tiny filesystem living in the Pico's flash, not your computer's
  4. Copies every file in firmware/ onto the device, preserving the same relative paths
  5. Soft-resets the device — equivalent to unplugging and replugging, but faster — which re-triggers the boot.py → main.py sequence above
Set expectations correctly for next time: once you've flashed MicroPython and run deploy.sh, if the screen isn't wired up yet, main.py will likely still run without raising an exception — spi.write() just shifts bits out a pin, it doesn't check whether anything is listening. "No error" will not mean "it worked." That's a real embedded-systems trap worth having named in advance, not discovered while debugging.

Check it stuck

Which of the two "uploads" requires holding a button on the board?
Which file does MicroPython always run first on startup, before main.py?
