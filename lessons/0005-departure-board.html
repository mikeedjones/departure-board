Lesson 5 · Putting it together

Building the departure board

Four lessons built the pieces in isolation: a screen that draws text over SPI, a radio that joins a network, an HTTP client that fetches JSON. This lesson wires them into one loop that polls a real API — RTT.io (Realtime Trains) — for live departures from a real station and renders them, on a timer, forever. It's also where the gap between "embedded" and "a full-powered machine" stops being theoretical: every section below is something that broke, or would have broken, specifically because this is a microcontroller and not a laptop.

A scope change worth naming: the reference implementation this lesson works from — chrisys/train-departure-display — talks to National Rail's OpenLDBWS, which is a SOAP/XML API. MicroPython has no XML parser in its standard library, and hand-rolling one is a lesson in itself, not a good first real-API integration. RTT.io covers the same UK rail data as plain JSON, so that's what the workbook actually calls — the camelCase-to-snake_case exercise below applies the same idea chrisys's script demonstrates, just against a JSON body instead of a SOAP envelope.

Discovery 1 — the key wasn't usable directly

The first thing the workbook tried was the obvious thing: put the RTT.io key straight in an Authorization: Bearer header and call the departures endpoint. It came back 401 {"error":"Invalid or expired token"}. Worth debugging on a laptop with curl, not by redeploying to the Pico over and over — a USB round-trip plus a Wi-Fi join is a slow way to iterate on an HTTP bug that has nothing to do with the board.

The key turned out to be a long-life refresh token, not an access token. RTT.io's spec is explicit about the two-tier model:

"You may either have: solely an access token which grants you a long-life entitlement to the system or, long-life refresh token which you will need to request a short-life access token with." Realtime Trains API specification

GET /api/get_access_token, authenticated with the refresh token, returns a real access token good for about 20 minutes. That access token is what actually goes on the departures request.

vs. a full machine: on a laptop, a library like requests-oauthlib or any modern SDK client hides this exchange-and-refresh dance behind one object you construct once. MicroPython has no such library here — get_access_token() in the workbook is the whole refresh flow, written out by hand, because there's nothing to import that does it for you.

Discovery 2 — no wall clock to check expiry against

The access token's response includes a validUntil ISO timestamp. The natural move is to compare it against the current time and refresh just before it lapses — except the Pico has no wall clock unless you explicitly sync one over NTP, which this project hasn't done. time.time() on a freshly-booted board is relative to boot, not to 2026.

The workaround is to stop trying to agree with the server's clock at all. time.ticks_ms() and time.ticks_diff() give a monotonic, boot-relative timer that's perfectly sufficient for "refresh again after roughly 15 minutes" — deliberately short of the ~20 minute real expiry, with margin, rather than exact:

ACCESS_TOKEN_LIFETIME_MS = 15 * 60 * 1000

def get_access_token(force=False):
    global _access_token, _access_token_fetched_at
    now = time.ticks_ms()
    if (not force and _access_token is not None
            and time.ticks_diff(now, _access_token_fetched_at) < ACCESS_TOKEN_LIFETIME_MS):
        return _access_token
    ...

Belt and braces: fetch_departures() also catches a 401 on the departures call itself and force-refreshes once before giving up. Between the timer and the reactive retry, an expired token never surfaces as a visible failure.

De-camel-casing a live API

RTT.io's JSON, like most web APIs, is written in camelCase — temporalData, scheduleAdvertised, realtimeForecast, locationMetadata. Idiomatic Python is snake_case. normalize_service() is the one function in the workbook whose entire job is that translation, flattening a deeply-nested camelCase tree into the flat dict the display actually renders:

departure = service["temporalData"]["departure"]
scheduled = departure["scheduleAdvertised"]
estimated = departure.get("realtimeActual") or departure.get("realtimeForecast") or scheduled
...
return {
    "scheduled": hhmm(scheduled),
    "estimated": hhmm(estimated),
    "is_cancelled": bool(departure.get("isCancelled")),
    "platform": platform,
    "destination_name": destination_name,
    "operator_name": operator_name,
}

Everything past this one function deals only in snake_case, board-shaped fields — render() never sees a camelCase key. That boundary is the same idea chrisys's OpenLDBWS script applies to SOAP/XML: parse the vendor's shape once, at the edge, into your own.

No datetime, either

RTT.io's timestamps are ISO 8601: "2026-07-02T18:32:00". On a full Python install, datetime.fromisoformat() parses that in one call. MicroPython's standard library doesn't ship a general-purpose datetime module at all — the workbook's hhmm() just slices the string:

def hhmm(iso_datetime):
    if not iso_datetime:
        return "--:--"
    return iso_datetime[11:16]

That's not a shortcut taken for this lesson specifically — it's genuinely the normal way to handle fixed-format timestamps in MicroPython when you don't need general date arithmetic. Pulling in a full ISO-8601 parser for six characters of a string would cost more flash and RAM than the problem is worth.

Memory is a shared, finite thing now

The RP2350 has 520 kB of SRAM total — code, the display driver's framebuffer-adjacent state, network buffers, and every JSON object you've parsed all share that one pool. response.json() materializes the entire nested response as live Python objects in one shot. A wide time window or an unfiltered station can return dozens of services, each a multi-level dict — on a laptop with gigabytes of RAM this is a non-event; here it's a real constraint, which is why the workbook narrows the query with filterTo rather than fetching everything and filtering in Python afterward.

The loop also calls gc.collect() explicitly after every poll:

# No pip, no persistent HTTP session, no OS-managed heap on this
# board — urequests opens a fresh socket every call, and the
# 520 KB of SRAM this shares with everything else means a forced
# collect after each poll is worth the cycles it costs.
gc.collect()
time.sleep(POLL_SECONDS)

On CPython you essentially never call gc.collect() by hand — the generational collector runs on its own schedule and there's enough headroom not to think about it. On a board with 520 kB total, forcing a collection right after the biggest allocation of the loop (the parsed JSON body, now out of scope) is cheap insurance against fragmentation slowly eating the heap over a device that's meant to run for weeks.

One loop, no threads

There's no background thread quietly polling while the display stays responsive. main() is a single while True: fetch, render, sleep, repeat. If fetch_departures() is slow — a sluggish DNS lookup, a slow TLS handshake — the display simply doesn't update until it returns. The workbook's answer isn't concurrency, it's graceful degradation: on any exception, render(display, [], error=e) puts the error on screen instead of leaving the board frozen on stale data with no explanation, and the loop keeps going next cycle rather than crashing out.

Secrets, in plain text, on the device

The refresh token lives in config.py on the Pico's own filesystem, unencrypted — the same pattern already used for the Wi-Fi password. There's no keychain, no environment-variable injection, no secrets manager between the filesystem and the value your code reads. RTT.io's own spec flags this as a real concern, not a hypothetical one:

"It is a requirement that no token is placed in a distributable user application unless specifically authorised by us. End-user applications are expected to proxy their requests through a server-side application such that token is not available publicly." Realtime Trains API specification

For a single personal device that never leaves your desk, that risk is low — but it's worth being honest about what "low" means here: anyone with physical access to this board and a way to read its flash gets the token in the clear. A server-side deployment would keep the token behind a proxy specifically so a compromised client never sees it. An embedded device generally can't do that, because the device is the client making the request directly.

Case study: an API that's honest about its own gaps

The workbook shows Plat TBC for a lot of real departures from Wandsworth Town — even though National Rail Enquiries' own site shows a platform for some of those same services. Checking /rtt/service (the full per-stop journey, not just the departures summary) for one of those trains confirmed it wasn't a parsing bug: RTT.io's data for that specific stop has no platform value at all, at any level of detail, while other stops on the same service (Waterloo, Clapham Junction, Wimbledon) all have one.

Two live systems drawing on the same underlying Network Rail data can genuinely disagree on completeness for a given field at a given station — NRE likely falls back to a static default platform assignment for small stations with a fixed platform-per-direction, where RTT.io only echoes explicit real-time allocation messages and none were ever sent for this stop. TBC is the honest thing to show, not a bug to chase — the same "read what the API is actually telling you, not what you assumed it would" instinct from reading the ILI9341 datasheet against its driver in lesson 2, now applied to a live third-party API instead of a chip.

Step — configure your station, deploy

The workbook's config.py now carries everything: Wi-Fi, display pins, and the RTT.io refresh token plus your origin/destination CRS codes:

RTT_IO_REFRESH_TOKEN = "your-refresh-token"
RTT_IO_HOST = "data.rtt.io"
STATION_CRS = "WNT"   # your departure station
DEST_CRS = "WAT"      # leave "" to show every destination
uv run mpremote cp workbooks/lesson-0005-departure-board/firmware/config.py :config.py
uv run mpremote cp workbooks/lesson-0005-departure-board/firmware/main.py :main.py
uv run mpremote cp workbooks/lesson-0005-departure-board/firmware/lib/ili9341.py :lib/ili9341.py
uv run mpremote reset
Win condition
The board connects, then shows up to six live departures from your station — scheduled time, destination, platform (or TBC), amber "exp HH:MM" when a service is running late, and red "cancelled" rows for cancellations — refreshing once a minute, indefinitely, recovering on its own from a dropped Wi-Fi link or an expired token.
vs. Arduino/C: the C SDK has no built-in JSON type at all — you'd reach for a third-party library like cJSON, walk a pointer-based tree, and manage its memory yourself. TLS is a similar story: MicroPython's urequests on this port already links against a TLS stack for you (certificate verification disabled, per lesson 4, but the handshake itself just works); on the C SDK you'd configure lwIP's mbedTLS integration by hand before your first HTTPS byte. Both gaps — JSON and TLS — are exactly where MicroPython's "batteries included" standard library earns its keep for a project like this one.
