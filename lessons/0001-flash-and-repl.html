Flashing MicroPython, and your first REPL

Parts are assembled: screen, Pico 2 W, breadboard, jump wires. None of that gets touched yet. Before any SPI wiring, the board needs an interpreter on it. That's this lesson: flash MicroPython, get a REPL, prove Python is actually executing on the chip.

What "flashing" actually does

On Arduino, avr-gcc/arm-none-eabi-gcc compiles your sketch to machine code, and the bootloader writes that binary to flash on every upload — one compile-and-flash cycle per code change. MicroPython inverts this: you flash a C-compiled interpreter onto the RP2350 once, and from then on your .py files are just text it reads and executes at runtime. No recompiling, no reflashing, for every line you change.

WhatHow often
Flash The MicroPython interpreter itself — a .uf2 image Once (or on a version upgrade)
Deploy Your .py files, copied onto the interpreter's filesystem Every code change
This lesson only does the first row. Deploying main.py at the end is a preview of the second — full detail in the workflow reference.

Step 1 — flash the interpreter

  1. Download the Pico 2 W build: micropython.org/download/RPI_PICO2_W
  2. Disconnect the board from USB. Hold BOOTSEL, plug the USB cable in, then release.
  3. A drive called RPI-RP2 mounts. Drag the downloaded .uf2 onto it.
"It should take a few seconds to program the UF2 file into the flash. The board will automatically reboot when finished, causing the RPI-RP2 drive to disappear, and boot into MicroPython." Raspberry Pi — Connecting to the Internet with Pico W-series, §3.2

Step 2 — get a REPL

Already a Python developer, so skip Thonny's GUI and use mpremote from the terminal instead — it's already added to this project (uv add mpremote, done).

uv run mpremote connect list   # find the port
uv run mpremote                # open a REPL — Ctrl-] to exit

At the >>> prompt:

>>> import sys
>>> sys.implementation
(name='micropython', version=(1, 2X, 0), ...)

That's the win condition for this step: a real Python interpreter, talking back, running on hardware that has no operating system underneath it.

Aside — the on-board LED isn't on the RP2350

Try this at the REPL:

>>> from machine import Pin
>>> led = Pin("LED", Pin.OUT)
>>> led.toggle()
"Unlike the non-wireless Pico-series, the on-board LED on Pico W-series is not connected to a pin on RP-series microcontroller, but instead to a GPIO pin on the wireless chip." Raspberry Pi — Connecting to the Internet with Pico W-series, §3.4

That's WL_GPIO0 on the CYW43439 — the same chip that will eventually do Wi-Fi. Worth noticing now, because it means Pin("LED") and Pin(2) are not interchangeable the way "the LED pin" is on an Arduino Uno: the string name routes through a completely different chip than a GPIO number would.

Step 3 — deploy your first program

The workbook for this lesson has firmware/main.py — five LED toggles and two print statements, nothing more. Copy it onto the board and run it:

uv run mpremote cp workbooks/lesson-0001-flash-and-repl/firmware/main.py :main.py
uv run mpremote reset

reset triggers a soft-reboot, and MicroPython automatically re-runs whatever's saved as main.py on the device — no REPL command needed to launch it.

"If boot.py is found then it is executed... a file named main.py... is executed next." MicroPython docs — Reset and boot

Rough Arduino mapping: main.py ≈ loop(), except it runs exactly once, top to bottom — no implicit repeat. There's no boot.py in this workbook yet; it's optional, and there's nothing that needs one-time init before main.py yet.

Win condition
The LED toggles five times and uv run mpremote (reconnect to see it) prints two lines. That's Python, executing on the chip you soldered nothing to yet.
