# Lesson 5 complete: real live departures on the board, refresh-token exchange debugged independently

Wired lesson 1-4's pieces into one working departure board: Wi-Fi connect, RTT.io poll, JSON parse, ILI9341 render, on a one-minute loop. The RTT.io key issued from api-portal.rtt.io turned out to be a refresh token, not a directly-usable access token — caught this by testing with `curl` against the live API before writing any device code, rather than discovering it through a failing on-device deploy. That instinct (verify the third-party contract on the dev machine, reserve the Pico for hardware-specific behavior) is now a standing preference, not just how this one bug got found.

Also independently flagged a real data-quality question mid-build: the board showing `Plat TBC` for stations where National Rail Enquiries shows a platform. Traced it to source via `/rtt/service`'s full per-stop journey rather than assuming it was a parsing bug — confirmed RTT.io genuinely has no platform value for that station on that service, at any level of query detail. Correctly read this as two live systems disagreeing on data completeness, not a bug to fix, and is planning to layer their own platform-inference logic on top later. Good sign for future work with imperfect real-world APIs: the "read what the API actually says, don't assume" habit from lesson 2's datasheet work transferred cleanly to a live JSON API.

No misconceptions to correct. The mission's core embedded-vs-full-machine contrasts (hand-rolled OAuth-style refresh with no library support, no wall clock without NTP so token expiry is tracked via `ticks_ms()` instead, no `datetime` module, explicit `gc.collect()` on a 520 kB shared heap, single-threaded poll-and-render with no background thread) are now all grounded in working code rather than abstract statements — this closes out the interleaved lesson arc MISSION.md describes.
