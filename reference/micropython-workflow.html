MicroPython on the Pico: flash vs. deploy

Compressed from Lesson 1. The one thing this exists to stop you forgetting: these are two different operations, not one "upload."

The two-layer model

WhatHow oftenHow
Flash the interpreter MicroPython runtime itself, a full .uf2 image Rarely — once, or on version upgrades Hold BOOTSEL while plugging in USB → drive mounts → drop the .uf2 on it
Deploy your code Plain .py files on the interpreter's filesystem Every code change scripts/deploy.sh (wraps mpremote) — no BOOTSEL

Boot sequence, every power-up / soft-reset

boot.py   → one-off init (runs once, every boot)
main.py   → your program (runs once, top to bottom; no auto-loop)
"If boot.py is found then it is executed... a file named main.py... is executed next." MicroPython docs — Reset and boot

Arduino equivalent: boot.py ≈ setup(), main.py ≈ loop() — except main.py does not repeat automatically, and there's no shared runtime state between the two beyond the filesystem.

Useful mpremote commands

mpremote                      # open a REPL on the board (Ctrl-] to exit)
mpremote cp file.py :file.py  # copy one file to the device
mpremote cp -r dir/ :dir/     # copy a directory
mpremote reset                # soft-reset (re-runs boot.py -> main.py)
mpremote fs ls :              # list files on the device

Why lib/ is vendored, not mip-installed

mip is MicroPython's package manager (pip's counterpart). This project copies third-party drivers straight into firmware/lib/ instead, so the exact version is committed and reviewable rather than fetched fresh on every deploy.
