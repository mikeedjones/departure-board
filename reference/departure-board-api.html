RTT.io (Realtime Trains) departures cheat sheet

Compressed from Lesson 5. Full spec: realtimetrains/api-specification.

Auth — two-step, token has a short lifetime

  1. You hold a long-life refresh token from api-portal.rtt.io.
  2. GET https://data.rtt.io/api/get_access_token, with the refresh token as Authorization: Bearer, returns a short-life access token (~20 min) plus a validUntil.
  3. Use the access token as Authorization: Bearer on every other call.
No synced wall clock on the Pico — track expiry with time.ticks_ms() / time.ticks_diff() against a conservative fixed interval (e.g. 15 min), not against validUntil. Also force-refresh once on any 401.

Endpoints used here

EndpointPurpose
GET /api/get_access_tokenExchange refresh token for access token
GET /rtt/location?code=gb-nr:XXX&filterTo=gb-nr:YYYDepartures summary from station XXX, filtered to destination YYY (omit filterTo for all destinations)
GET /rtt/service?uniqueIdentity=...Full per-stop journey for one service — more detail than the location summary, e.g. every calling point's own platform

Codes are namespace-prefixed: a 3-letter CRS code like WAT becomes gb-nr:WAT in a query.

Response shape → board fields

RTT.io's JSON is camelCase; normalize to snake_case at the edge, once, in one function:

RTT.io field (camelCase)Board field (snake_case)
temporalData.departure.scheduleAdvertisedscheduled
temporalData.departure.realtimeActual / realtimeForecastestimated (fall back to scheduled)
temporalData.departure.isCancelledis_cancelled
locationMetadata.platform.actual / plannedplatform (fall back to "TBC" — often genuinely absent, see below)
destination[].location.descriptiondestination_name (join with " & " if the service splits)
scheduleMetadata.operator.nameoperator_name

Timestamps are ISO 8601 without a timezone suffix in practice (e.g. "2026-07-02T18:32:00"). No datetime module in MicroPython — slice [11:16] for HH:MM rather than parsing.

Platform can legitimately be missing: confirmed via /rtt/service that some stations (small stops with a fixed platform per direction) never get an explicit real-time platform-allocation message, at any level of query detail — even though a site like National Rail Enquiries may still show a default platform for the same service. TBC is the honest fallback, not a bug.

Status codes worth handling

200Success, parse body["services"]
204Valid query, no services in the time window — not an error, treat as an empty list
401Access token expired — force a refresh and retry once
429Rate limited — response includes Retry-After and X-RateLimit-* headers per dimension (minute/hour/day/week)
